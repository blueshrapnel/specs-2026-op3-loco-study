\begin{abstract}
This extended abstract studies whether the MuJoCo Playground zero-shot sim-to-real recipe for humanoid joystick locomotion extends to the smaller ROBOTIS OP3 platform. MuJoCo Playground treats OP3 as a first-class humanoid environment alongside Berkeley Humanoid, Unitree H1/G1, and Booster T1, using the same joystick command abstraction of forward velocity, lateral velocity, and yaw rate. However, the paper and RSS~2025 demonstration evidence report real-hardware deployment on larger research platforms, not OP3. Our contribution is therefore not a new reinforcement-learning algorithm, but an engineering audit of the path from a canonical Playground OP3 policy to an OP3-facing deployment stack: MJX/Brax PPO training, actuator-order alignment with ROBOTIS motor IDs, ONNX export, ROS~2 policy execution, MuJoCo-to-Webots sim-to-sim testing, and measurement-driven sim-to-real diagnosis. The central claim is that small mismatches in observation conventions, command filtering, actuator stiffness, and simulator contact assumptions can dominate whether a policy that walks in simulation remains meaningful when moved into a second simulator or onto hardware.
\end{abstract}

\noindent\textbf{Keywords:} ROBOTIS OP3; MuJoCo Playground; Webots; ROS~2; humanoid locomotion; sim-to-sim; sim-to-real; reproducibility.

\section{Motivation}

High-throughput reinforcement learning can produce visually convincing humanoid locomotion in simulation, but transferring such policies to a physical robot depends on whether the training simulator, exported policy, middleware, and robot control stack describe the same closed-loop system. MuJoCo Playground is a useful test case because it is both a canonical implementation and a strong sim-to-real claim: the authors describe zero-shot transfer across six robot platforms in under eight weeks, with deployment through ONNX Runtime at 50~Hz, \texttt{ros2-control}, C++ real-time control, and a model-based estimator \citep{zakka2025mujocoPlayground,mujocoPlaygroundSite,mujocoPlaygroundDemo2025}.

The ROBOTIS OP3 is present in the Playground humanoid task family, but it is not reported as one of the real-hardware zero-shot platforms in the paper or RSS~2025 demo. This makes OP3 a productive SPECS case study: if the shared joystick abstraction transfers, OP3 provides evidence that the Playground recipe generalises to hobbyist-tier humanoid hardware; if it fails, the failure modes are still informative because they expose deployment assumptions in a widely used canonical library. Our goal is deliberately conservative: reproduce the reference walking behaviour, adapt it to a ROBOTIS-ordered STRIDE pipeline, and identify fidelity gaps before making strong hardware claims.

\section{Pipeline}

The current pipeline is:
\[
\text{MuJoCo MJCF} \rightarrow \text{MJX} \rightarrow \text{Brax PPO} \rightarrow \text{ONNX} \rightarrow \text{ROS 2} \rightarrow \text{Webots / OP3}.
\]

The robot is first defined in MuJoCo using an MJCF model, including kinematics, inertial parameters, contact geometry, and position-controlled actuators. This model is executed in MJX for accelerated physics simulation, enabling large-scale reinforcement learning with PPO implemented in Brax. The policy runs at 50~Hz with a 147-dimensional observation formed from three stacked 49-dimensional frames. Actions are 20-dimensional joint-position offsets around the default pose. A key implementation detail is actuator ordering: the STRIDE model reorders the MuJoCo actuators to match ROBOTIS motor IDs 1--20, reducing the chance that learned actions are permuted during deployment. The trained policy is exported to ONNX format to enable deployment outside the JAX runtime. A ROS2 node loads the ONNX model and performs real-time inference.

\section{Sim-to-Sim Transfer}

Before treating hardware experiments as sim-to-real evidence, we use Webots as an intermediate deployment rehearsal. The purpose is not to claim that Webots is more accurate than MuJoCo, but to test whether the exported ONNX policy and ROS~2 policy node can drive an OP3 model outside the original training simulator. This stage exercises the same observation builder, joint-state interface, IMU handling, command velocity input, action history, motor ordering, update timing, and position-command semantics intended for hardware.

The main comparisons will be commanded versus achieved base velocity, torso pitch and roll, foot contact timing, joint position tracking, action saturation, and qualitative failure modes such as foot scuffing, collapse, or turning bias. Disagreement between MuJoCo and Webots is useful when it localises a deployment assumption that was hidden in the training loop.

The Webots controller is a C++ ROS~2 node running a 20\,ms timer callback: read joint states and IMU, assemble the 147-dimensional observation, run the ONNX forward pass, write 20-dimensional joint position targets.  A gating counter enforces the 50\,Hz policy rate independently of the ROS~2 spin rate.  Command velocity is received on the standard \texttt{/cmd\_vel} topic via \texttt{teleop\_twist\_keyboard}, making the node API-compatible with the planned hardware deployment.

Several latent bugs were exposed during integration that would otherwise have appeared only on hardware.  (i)~\emph{IMU axis convention:} the Webots gyroscope axis mapping was determined empirically by perturbing the robot along each axis and inspecting raw sensor output; assuming documentation accuracy would have left a silent sign error. (ii)~\emph{Gravity vector transpose:} \texttt{getUpVector()} returns the third \emph{row} of the rotation matrix; using the third column produced a transposed gravity estimate and immediate falls. (iii)~\emph{Velocity-limit persistence:} a \texttt{setVelocity()} call made during initialisation persists across subsequent \texttt{setPosition()} calls, causing unrestricted-speed position commands until the limit is explicitly reset. (iv)~\emph{Timestep-dependent gain scaling:} training used $K_d=1.084$ at a 4\,ms physics step; Webots runs at 8\,ms, requiring $K_d=0.2$ for comparable joint damping.

\section{Preliminary Findings}

Initial reproduction work shows that matching the reference implementation matters more than changing the learning algorithm. Stable walking required Playground-compatible reward scaling, body-frame velocity tracking, observation normalisation, three-frame history, command sampling, and PD gain setup. A custom PPO implementation produced stable standing but poor walking until these details were brought into alignment.

Deployment testing has also exposed control-path issues. The ROBOTIS \texttt{direct\_control\_module} can low-pass filter position commands through minimum-jerk smoothing. For a policy emitting actions every 20~ms, these filtering parameters determine whether the robot receives near-passthrough policy commands or smoothed trajectories with substantially different timing.

\section{Sim-to-Real Fidelity Questions}

The main simulator mismatch identified so far is actuator stiffness. The default MuJoCo Playground OP3 stiffness is much softer than measurements from the physical robot in a static walk-ready pose. Action-page replay also revealed that trajectory shape matters: linear interpolation in simulation did not match the trapezoidal velocity profile used by the ROBOTIS action module. Foot-ground interaction is another open question, because the physical OP3 feet include protruding screws that can behave like high-friction contact points on outdoor surfaces.

\section{Contribution}

The contribution of this extended abstract is an engineering audit rather than a new locomotion method. We document a reproducible path from a public MuJoCo Playground OP3 task to a ROS~2 deployment stack, add a Webots sim-to-sim stage to test deployment assumptions, and identify the main fidelity gaps that must be resolved before robust physical walking claims are credible. The scientific focus is whether a first-class but not yet reported OP3 Playground environment can support the same zero-shot or near-zero-shot deployment narrative demonstrated on larger platforms.
